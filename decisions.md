## 2026-07-12 — Chose a public repo over private

Competition entry with no secrets in the tree and no reason to hide the work; public is simpler and the default for a portfolio/competition artifact. (journal/2026-07-12-1.md)

## 2026-07-12 — Framed the first Milestone as "Spec complete" rather than a build target

Stack, commands, and the crux-identification approach are all undecided, so committing to a build deliverable now would be guessing. Made the first milestone about turning spec.md into the design of record (Issues #1–#4) instead. (journal/2026-07-12-1.md)

## 2026-07-12 — Scoped to the "egg class" of ingestible-health questions, not all three competition cases

The tool's distinctive move is forward-looking (crux → pending evidence → pre-registered update rule). COVID-origins and LHC black holes are retrospective/resolved — no decisive evidence is incoming, so the "watch these spaces" step has nothing to grab. Eggs, multivitamins, artificial sweeteners, etc. have live cruxes with registered ongoing trials. Chose to argue generalizability *across this class* rather than across the three provided cases — narrower but demonstrable.

## 2026-07-12 — Deliverable is a Claude skillset, not a Python pipeline

Considered "prototype tool" as a standalone LLM pipeline (Python + Anthropic SDK + PubMed/ClinicalTrials.gov). Chose instead a set of Claude skills: hand Claude the skills + a question and it runs the process end to end. It is both a competition "specification" (transparent, steerable workflow) and a runnable "prototype," needs no install, and matches the project's markdown-first premise. Core is tool-agnostic; tools (ClinicalTrials.gov, PubMed/Europe PMC, Scite) are *suggested* enhancers, with Claude web search as the floor.

## 2026-07-12 — Live, uncached retrieval

Every run queries literature/trial sources fresh so the watch-list reflects current reality — favoring the "living document" selling point over reproducibility. Accepted tradeoff: non-reproducible runs and exposure to API downtime during judging. A caching layer is parked in *Future direction*.

## 2026-07-12 — ClinicalTrials.gov API as the spine of the "pending evidence" step

Its machine-readable list of ongoing/upcoming trials with expected completion dates turns "check back in six months" from a guess into a real query. This data source is what makes the forward-looking step feasible and is the reason the egg-class scope is the right fit.

## 2026-07-12 — Scope defined by two properties, not by subject (nutrition is just the example)

The question class is anything **unresolved** *and* **fast-moving in new study evidence** — not "nutrition." Nutrition/ingestible-health is the most legible example set and where the worked example lives, but the workflow targets any question with live, imminently-resolving cruxes. This keeps generalizability honest while still giving a concrete demo.

## 2026-07-12 — Operationalize step must never demand info the user lacks

The skill proposes operationalizations from the literature itself, picks the load-bearing framing, states its assumption, and only *offers* user input as an option — never gates on it. Guards against a tool that interrogates users about outcomes/doses/subgroups they can't be expected to know.

## 2026-07-12 — Two non-negotiable build commitments: self-containment and calibrated humility

(1) The worked example is produced by a fresh Claude in a folder containing only the skills, so everything the prototype needs must live in the skill files — this is the build's primary correctness test and moots the skill-granularity question (whatever we pick must pass a cold-start run). (2) The process must know what it doesn't know: fail-state is explicit refusal / `insufficient-evidence`, never fabricated cruxes or trials. Schema carries first-class `unknown`/`insufficient-evidence` values and a `gaps[]` list — including "a live crux nobody is studying," which the humility requirement turned into a valuable output rather than a hole.

## 2026-07-12 — Prior-art borrows chosen per step (GRADE+RoB2, VOI/EVPPI, Metaculus conditional pairs)

From a prior-art scan. Step 1 validity uses **GRADE + RoB 2** (RoB 2 chosen as the most LLM-automatable appraisal tool — a signaling-question decision tree applied deterministically; AMSTAR-2/ROBIS for reviews, flagged lower-confidence). Step 2 cruxes are framed as **Value of Information / EVPPI** (the quantitative formalization of "crux") plus double-crux as the plain-language framing. Step 4 update rules borrow **Metaculus conditional pairs** + GRADE Evidence-to-Decision + Bayesian likelihood ratios. Living Systematic Reviews are cited as the watch-list's lineage, with their sustainability failure (half of Cochrane's COVID living reviews never updated) as our reason-to-exist. Rationale: don't reinvent evidence appraisal/forecasting; borrow-with-credit and position against them.

## 2026-07-12 — Skills depend on concepts/frameworks only, never on an external API or tool

Tool-agnosticism promoted to a hard rule and merged with self-containment: no skill may *depend* on ClinicalTrials.gov, Semantic Scholar, Scite, or any third-party API/tool. The frameworks (GRADE, RoB 2, VOI, etc.) are built into the skills as reasoning procedures in our own words; specific tools are named only as optional accelerators, and the workflow must run to completion on Claude's built-in web search alone. Rationale: (1) portability — the worked example runs in a bare folder; (2) it is our licensing firewall — depending only on concepts we author keeps us clear of third-party terms of use (Borrow Type B). Reframed ClinicalTrials.gov from "spine of step 3" to "fastest access point to the registry-lookup concept, not a requirement."

## 2026-07-12 — Licensing audit cleared all borrows; encoded the operative rules in the skills

Audited every prior-art borrow for a public open-source entry. Nothing disqualifying. Anchor: 17 U.S.C. § 102(b) — methods/procedures aren't copyrightable, only their specific expression, so describe-and-apply in our own words is clear. Operative constraints now in spec.md *Permissions & attribution*: (1) required citations incl. AMSTAR-2 attribution which is mandatory (CC BY); (2) do NOT paste RoB 2 signaling-question text (CC BY-NC-ND) or ROBIS items or substantial Double Crux post text verbatim — AMSTAR-2 verbatim is the one exception; (3) trademark cautions — never label output "Superforecaster/Superforecasting" (Good Judgment ®; use "calibrated forecasting"), don't imply Cochrane endorsement, don't call a deviating adaptation "GRADE"; (4) API courtesy lines apply only if a user opts into a tool — since the skills depend on no API, no third-party terms bind the prototype. Reinforces the tool-agnostic decision: not depending on APIs is also the licensing firewall.

## 2026-07-12 — Skills detect available tools at runtime and match them to roles

The orchestrator and every retrieval-bearing stage skill check at runtime which tools are actually present (MCP servers, connected APIs, web search) and use the most capable relevant one, degrading to web search. Skills name the *role* they need ("trial-registry lookup," "citation-stance data") rather than hard-coding a tool. Keeps tool-agnosticism real while still exploiting tools when a user has them — tool absence changes thoroughness, never whether a step runs.

## 2026-07-12 — Adapt RoB 2 by borrowing its mechanism, not rephrasing its questions

Decided the step-1 validity appraisal reuses RoB 2's *mechanism* — 5-domain skeleton, the Yes/Probably-Yes/Probably-No/No/No-Information response schema, and the factual-signaling-questions → deterministic-algorithm → Low/Some-concerns/High pipeline — but authors our **own** signaling questions from an understanding of each domain's bias mechanism rather than paraphrasing Cochrane's (which also respects the CC BY-NC-ND no-derivatives limit). Two payoffs beyond legality: RoB 2's "No Information" response is a ready-made humility valve that maps onto our `insufficient-evidence`/`gaps[]` states; and because RoB 2 grades a *specific result*, appraisal lives inside the crux structure (each study judged relative to the crux it bears on), and step 0 inherits the assignment-vs-adherence (ITT vs per-protocol) distinction. Rationale: the value of RoB 2 is the division of labor (LLM answers factual questions, fixed algorithm aggregates), not the specific wording — so rephrasing would copy the wrong thing.

## 2026-07-12 — Compounding = incorporable into a larger project; we do NOT build the updating loop

Editor pass caught the spec selling the artifact as "re-runnable, mechanically re-checked" without any workflow step that consumes a prior artifact and applies its update rules — an over-promise. Resolved by scoping *down*, not building up: our deliverable is a **one-shot artifact** with pre-registered update rules, designed to be **folded into someone else's larger epistemic project** (a personal knowledge base, an ongoing review). Maintaining/re-running/auto-updating a watch-list over time is explicitly out of scope (the "living database" is the user's project). This keeps the compounding claim honest and keeps the 7-day build tractable.

## 2026-07-12 — Walk back rigor we don't have (VOI and Rootclaim borrows are qualitative)

Editor pass flagged two overclaims that brushed our own calibrated-humility rule. Fixed: (1) VOI/EVPPI is borrowed as a **qualitative reasoning lens** to rank cruxes, not a numeric computation — we have no quantified decision model and claim no EVPPI number. (2) Rootclaim's likelihood-ratio→posterior is borrowed as the **idea** of direction/strength weighting behind step 4's "update toward X by rough magnitude," not the full numeric LR math. Principle: never imply more rigor than the tool delivers. Also fixed a stale contradiction ("we query existing public APIs" in Out of scope) left over from before the tool-agnostic pivot, plus redundancy/polish.

## 2026-07-12 — end-session always pushes after committing

Chose to make `git push` a mandatory final step of the end-session skill rather than leaving it optional/manual. A session isn't closed until its work is on the remote; leaving commits local-only is how a "closed" session silently isn't. On push failure the skill retries once then asks the user to run `! git push` — the commit stands, only the push defers. (Prompted by a network-blocked push from the agent environment this session.)

## 2026-07-12 — The defensible contribution is the loop, not any single step

The scan found no existing project spanning crux → pending evidence → pre-registered update rule; the pieces live in silos (living reviews own 1+3, Metaculus owns 4's structure, SCORE owns claim credibility, VOI is unused glue). Our novelty is connecting them: VOI-ranked cruxes → registry-driven pending-evidence tracking → pre-committed conditional updates against a GRADE-appraised base. The writeup leads with this positioning so judges see contribution, not reinvention.

## 2026-07-12 — Watch-list schema lives in the orchestrator skill, not a standalone doc

Building #12 (artifact serialization), decided the concrete schema's single home is a `## Artifact schema` section inside the orchestrator skill (`.claude/skills/crux-watchlist/SKILL.md`), not a `schema/` doc. Two reasons: (1) self-containment — a fresh Claude in a folder of only the skills must have the schema, so it must ship inside a skill file; a repo-root doc would either be absent at runtime or become a second home; (2) locality — step 5 of the pipeline *is* "assemble the artifact," so the schema belongs to the skill that emits it. spec.md keeps the *narrative* field list + design intent and now points to the skill for the concrete format (two different facts, one pointer, no duplication). #12 delivers only the schema section; the orchestration logic is stubbed for #5 to fill in the same file. Validity-`grade` and watch-`status` enums left as flagged placeholders — owned by #7/#8 and #10. Provisional: skills live under `.claude/skills/` (formalized by #16).

## 2026-07-12 — operationalize declares the estimand and defaults to ITT/real-world; stage skills own artifact sections

Building #6 (operationalize). Two calls beyond the spec's letter: (1) The skill makes the RoB 2 assignment-vs-adherence distinction an explicit **estimand** field on every operationalization, and **defaults to assignment/ITT (real-world intake)** — the effect a typical asker's decision needs — carrying per-protocol only when a crux plausibly turns on it. For observational questions it names the **target trial** the cohort emulates. This fixes "which effect is bias" *before* step 1 appraisal runs, which is the whole point of inheriting the distinction at step 0. (2) Established the pattern that **each stage skill owns one section of the artifact** — operationalize owns `# Question` — so the schema (#12) is the shared contract and skills compose without overlapping homes. The never-gate rule is implemented as "offer a re-run in the *output*, never return a question to the user," and is the skill's stated acceptance test.

## 2026-07-12 — appraise-study: RoB 2 mechanism as a standalone component, own questions, non-RCT gate

Built #8, the meatiest piece, as its own composable skill (`.claude/skills/appraise-study/`) rather than folding it into lit-review — it's invoked per (study × crux) and benefits from independent reuse/improvement. Key calls: (1) **Authored our own signaling questions** from each domain's bias mechanism (respecting RoB 2's CC BY-NC-ND on the question *text*); we kept the 5-domain skeleton, the Yes/PY/PN/No/No-Information response set, and the factual-Qs→fixed-algorithm division of labor — all method, not expression. Each question carries an explicit `low_risk_when` so the model only answers near-factual questions and never sets the grade (the acceptance test). (2) **Authored our own deterministic aggregation** (per-domain rules → overall = worst domain) rather than copying RoB 2's crosswalk tables — keeps it fixed and ours. (3) **`No Information` → `some-concerns` + gap flag, never `low`** — an under-reported trial can't earn a clean pass; unjudgeable domains route to the artifact's `gaps[]`. (4) **Non-RCT gate:** cohorts/case-control (most nutrition evidence) get Domains 3–5 + a single confounding check, capped at `some-concerns`, with body-of-evidence certainty deferred to lit-review's GRADE step (#7) — honest scoping over forcing an RCT instrument onto observational data. This finalizes the schema's per-source validity `grade` enum as `low | some-concerns | high` (was a #7/#8 placeholder in #12); GRADE body-certainty stays #7's. Skill carries the required Sources & methods block (Sterne et al., BMJ 2019;366:l4898; Cochrane non-endorsement; not-branded-GRADE; trademark line).

## 2026-07-12 — lit-review: grade the root claim in step 1, defer per-crux grading to step 2

Building #7. Surfaced a sequencing tension: `appraise-study` grades a study *relative to the crux it bears on*, but cruxes aren't identified until step 2 (#9). Resolved by decomposing the appraisal moment: **step 1 invokes `appraise-study` against the primary operationalized question as the root claim** (yielding the body-certainty behind `current_answer`), and **step 2 re-invokes the same crux-agnostic component per crux** to fill each `current_evidence`. This keeps "appraisal lives inside the crux structure" honest without requiring cruxes to exist prematurely, and gives step 1 a crisp deliverable: `# Current answer` + a *graded evidence base with the disagreement located* (the candidate cruxes handed to #9). Other calls: body certainty is **GRADE-based, not GRADE-branded** (apply the five downgrade domains in our words; cite the Working Group; honor the "don't badge as GRADE if we deviate" rule). Confidence mapping fixed: High→high, Moderate→moderate, Low/Very-low→low, with **`insufficient-evidence` reserved for when no best-estimate direction is responsibly supportable** (not a lazy fallback). Added provenance QA (SemanticCite 4-class: Supported/Partial/Unsupported/Uncertain) to catch claims whose cited source doesn't actually back them, and a standing "keep a human in the steering loop" note grounded in the SCORE (scoring ≈ chance) and hybrid-beats-automated findings. AMSTAR-2/ROBIS for review-appraisal, flagged lower-confidence.

## 2026-07-12 — find-cruxes: qualitative VOI operationalized as leverage × liveness; per-crux grading lands here

Built #9. Operationalized the "qualitative VOI, not numeric EVPPI" borrow as an explicit **two-factor crux test**: a sub-question is a crux to the degree it has both **leverage** (resolving it each way materially changes the answer — the swing distance) and **liveness** (genuinely unresolved now, not already settled by high-certainty evidence). Both required — high leverage but settled = background (no VOI); live but low leverage = trivia. This makes "which unknown most changes the conclusion" a checkable procedure rather than a vibe, while claiming no EVPPI number. Double crux is the plain-language name (described, not reproduced — LessWrong text). Confirmed the #7 hand-off: **this is where per-crux appraisal happens** — find-cruxes re-invokes `appraise-study` per crux, relative to that crux, to fill `current_evidence`. Kept the gaps discipline sharp: a *suspected* crux where neither leverage nor liveness can be shown from evidence goes to `gaps[]`, never into the list dressed as established; a real-but-ungradeable crux (no evidence bears on it yet) also goes to gaps and is exactly what step 3 hunts. Explicit anti-quota rule: never pad the crux list — a fabricated crux propagates into a fabricated watch-list item and update rule (three steps of confident fiction from one padded list).

## 2026-07-12 — find-pending-evidence: tool-detection pattern implemented here; relevance gate + no-fabricated-IDs

Built #10, the crux → pending-evidence half of the novel seam. Calls: (1) The runtime tool-detection pattern (name the *role* — "a lookup of registered ongoing studies + report dates" — detect what's available, use the most capable, degrade to web search; absence changes thoroughness not whether the step runs) is **implemented concretely here** because step 3 is the first real retrieval step, with an explicit note that cross-cutting #13 will generalize it across stage skills. Kept a "live, uncached, best-effort coverage, disclosed not hidden" stance consistent with the spec's retrieval policy. (2) **Hard relevance gate:** a study only makes the watch-list if what it *measures would actually move its crux* (an LDL trial doesn't resolve a CVD-events crux); `measures` must state the mapping, which is also the hook step 4 writes update rules against. Same anti-padding discipline as find-cruxes. (3) **Never fabricate a registry ID** — an invented NCT number is the worst failure (a fictional promise of resolution); unverifiable studies are dropped, not listed. (4) A crux with no pending evidence found → `gaps[]` as a first-class finding ("live crux nobody is studying"), never a silent empty cell. Finalized the schema's `status` enum (`recruiting | active | completed-unpublished | reported | terminated | withdrawn | unknown`) — with #8 (grade) and #7 (confidence) that clears the last of the schema's placeholder enums; all three now owned by their producing skills.

## 2026-07-12 — write-update-rules: mandatory "inconclusive" branch, qualitative magnitude bands, data-driven recheck

Built #11, completing the seam (crux → pending evidence → pre-registered update rule). Calls: (1) **Every rule must include the "it didn't really answer it" branch** — inconclusive / underpowered / terminated / never-reported → *no update* (+ gaps note). Writing only "confirms vs refutes" quietly launders the common ambiguous-readout case into a fake signal; the third rail is non-optional. (2) **Magnitude is a qualitative band** (`negligible | small | moderate | large`), never a numeric likelihood ratio — Rootclaim's direction/strength idea as *reasoning* only (cited with its public failure), Metaculus conditional-pair *structure* only. Magnitude is calibrated to two things already in the artifact: the crux's leverage (`why_it_moves`) × the watched study's weight (design/appraised quality) — a rule promising a `large` update from a study too weak to deliver caps itself, or it's an overclaim. (3) **`recheck_on` is data-driven** — the earliest meaningful `expected_report_date` on the watch-list, not an arbitrary interval; falls back to a bounded default only when all dates are `unknown`, and says so. (4) **Bias check:** if every branch of a rule points the same way, the rule is broken — it encodes a lean, not a conditional. Owns per-crux `# Update rules` + top-level `recheck_on`. All 5 stage skills (0–4) now built; only the orchestrator (#5) remains to wire them.

## 2026-07-12 — crux-watchlist orchestrator: end-to-end contract trace clean; three conductor-only threading duties

Built #5, filling the orchestration stub in the same file that already held the schema (#12) — the whole functional core (schema + 5 stage skills + appraisal component + conductor) is now standing. Wrote it after an end-to-end paper trace of the contract chain (the in-context integration check promised earlier, since the honest behavioral test must be clean-room per #19). The trace found the stages thread cleanly and surfaced three obligations that *only* the orchestrator can satisfy, now encoded explicitly: (1) **Estimand threading** — operationalize declares assignment/ITT vs adherence/PP on the primary framing, and it must reach *both* appraise-study invocations (lit-review's root-claim appraisal AND find-cruxes' per-crux appraisal) because Domain 2 branches on it; cruxes inherit the primary framing's estimand. This wasn't explicit in any single stage skill — it's a genuine integration requirement the trace caught. (2) **Gaps aggregation** — every stage emits gaps; the conductor collects them into one `# Gaps` section, deduped, each traceable to its step. (3) **Degrade-don't-crash** — a refusing stage (non-empirical question; no usable evidence) stops threading but still yields a well-formed, mostly-gaps artifact; refusal is a success path, not an abort. Also: single up-front tool-detection pass carried into retrieval stages; runtime date-stamping of `as_of`; crux-centric assembly (watch-list + update-rules nested under each crux); pre-emit validation (every crux_ref resolves, every field present-or-explicitly-unknown). Acceptance restated in-skill: runs cold, tool-agnostic, honest-by-construction.

## 2026-07-13 — validation strategy: soft question first (cold-run + honesty gate), hard question reserved as the generalization test

First behavioral tests, run clean-room (this session was command-center only). Chose to validate on **omega-3/CVD** — a *soft* strip — before the harder **aspartame** case, and ran the pre-registered **A′ framing-matched control** alongside Run B. Rationale, sharpened by the control's result: a clean pass on a soft, in-training, trial-shaped question proves the skills **run cold, tool-agnostic, and refuse honestly** (Run B: 0/5 predicted failures; refused to fabricate the missing C1 trial) — but it *cannot* prove the method beats base Claude, because base Claude already knows the question. The A′ control confirmed exactly this: un-scaffolded Claude **ties on cruxes and the central honest refusal**, but loses on **rigor/verification** (zero fetches, an observational/surrogate ID mis-assigned as a resolver, an already-reported trial filed as pending, numeric false-precision confidence, no systematic grading). So the defensible claim from a soft question is narrow — *the method makes output verified and auditable, not that it finds better cruxes* — and the generalization claim must be earned on a **hard** question where base Claude's recall advantage disappears (aspartame: regulatory + observational, non-RCT-shaped). Consequence: **no skill fix warranted from A′** (its failures are what the skills prevent), and the aspartame strip is prioritized over more soft cases. Also enforced Run B/A′'s web-search floor by **disconnecting Scite at the account level** rather than trusting its unauthenticated-cache state — belt-and-suspenders for the honesty test; reconnect before any tooled Run C.
